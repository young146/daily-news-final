# 번역 실패 관리 방안

> 뉴스 본문 번역이 실패했을 때 **어떻게 막고 · 어떻게 저절로 낫고 · 어떻게 손으로 정리하는지** 정리한 운영 기준. (2026-07-03 정립)

## 대원칙 — "실패는 완료가 아니다"

예전 사고: 번역 실패 시 제목·요약·본문 3칸에 `"Translation Failed"`를 저장했고, 배치 번역이 이걸 *"3칸 다 참 = 완료"* 로 오인해 **영구 스킵** → 27건이 DRAFT에 눌러앉음. 어떤 버튼으로도 다시 안 돌아가는 상태였음.

→ **이제 실패는 절대 "완료(DRAFT)"로 저장하지 않는다. PENDING(대기)으로 남겨 반드시 다시 시도되게 한다.**

## 3층 방어

### 1층. 예방 — 실패 자체를 줄임
- **동시 호출 10→5**, 배치 간 대기 200→1000ms → 순간 과부하(rate limit) 완화
- **재시도 3→5회, 지수 백오프+지터, `Retry-After` 헤더 준수** → API가 "몇 초 뒤"라고 하면 그대로 기다렸다 재시도
- 관련 코드: [lib/translator.js](../lib/translator.js) `computeBackoffMs`, [app/api/batch-translate/route.js](../app/api/batch-translate/route.js), [app/admin/actions.js](../app/admin/actions.js) `batchTranslateAction`

### 2층. 자동 복구 — 실패해도 저절로 나음
- 실패한 항목은 DB에 결과를 저장하지 않고 **`translationStatus: PENDING`** 으로 되돌림
- 다음에 번역을 다시 돌리면(워크플로 "번역 & 요약 생성" 버튼 등) **자동으로 재시도됨**
- 스킵 판정에서 `"Translation Failed"` 저장분은 제외 → 혹시 남아도 다음 배치에 재시도
- 판정 기준: `isFailedTranslation()` / `isStoredFailedTranslation()` ([lib/translator.js](../lib/translator.js))

### 3층. 수동 정리 — 쌓였을 때 한 방에
```bash
npm run retry-failed
```
- DRAFT 중 `"Translation Failed"`로 눌러앉은 항목만 찾아 **순차(0.5초 간격)로 재번역**
- 성공 → 정상 DRAFT / 또 실패 → PENDING / 원문 없음 → PENDING(수동 확인 대상)
- rate limit 안 나는 안전한 속도. 실패분만 손대므로 다른 데이터 무관
- 스크립트: [scripts/retry-failed-translations.js](../scripts/retry-failed-translations.js)

## 운영자 가이드 (사장님용)

**Q. 번역 실패한 기사가 보이면?**
1. 몇 개 안 되면 → 그 기사 번역 페이지 열고 **"AI 번역 생성"** 버튼 다시 누르면 됨
2. 여러 개면 → 터미널에서 `npm run retry-failed` 한 번 실행 → 알아서 다 복구

**Q. 실패가 몇 개인지 어떻게 아나?**
- 일괄 번역 후 뜨는 알림에 `실패: N개` 표시됨
- 또는 `npm run retry-failed` 실행 시 맨 위에 `대상 N건` 표시(0이면 깨끗)

**Q. 재실패(PENDING)로 빠진 건?**
- 대부분 원문 자체가 부실하거나 일시적 문제. `npm run retry-failed` 를 나중에 한 번 더 돌리면 대개 해결. 계속 실패하면 그 기사만 수동 확인/삭제.

## (선택) 자동화
지금은 실패가 급감하므로 수동 `npm run retry-failed`로 충분. 만약 완전 자동으로 돌리고 싶으면, 주간 인기검색어(weekly-keywords.yml)처럼 GitHub Actions로 주 1회 `retry-failed`를 돌리게 할 수 있음. (GitHub Secrets에 `ANTHROPIC_API_KEY`·`DATABASE_URL` 필요)
